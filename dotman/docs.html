Documentation

Welcome to the official documentation for dotman. Dotman is a high-performance dotfiles manager written in Nim, designed with Data Oriented Design principles for maximum efficiency.

Unlike GNU Stow or other dotfile managers, dotman features multi-profile support, automatic conflict detection, and a zero-configuration approach. It works seamlessly on Linux, macOS, BSD, and Windows.

Installation

Dotman is built with Nim and can be installed via several methods. It works on Linux, macOS, BSD, and Windows.

Install from Source

Clone the repository and build the binary:

git clone https://github.com/yourusername/dotman.git
cd dotman
nimble build
nimble install

Install via Nimble

If you have Nim installed:

nimble install dotman

Install via AUR (Coming Soon)

paru -S dotman

Requirements:

  • Nim 2.x or later
  • GCC or Clang (for compilation from source)

Via Nimble

If you have the Nim compiler installed:

nimble install dotman

Quick Start

Once installed, initialize dotman and start managing your configuration files in minutes.

# Initialize dotman (creates ~/.dotfiles)
dotman init

# Create the main profile (if not already created)
dotman make main

# Add your nvim config (moves file and creates symlink)
dotman set ~/.config/nvim

# Add your fish shell config
dotman set ~/.config/fish/config.fish

# Check status of your configs
dotman status

Directory Structure

By default, dotman uses ~/.dotfiles as the root directory. The structure is organized by categories that map to specific system locations:

~/.dotfiles/
└── main/ # Default profile
    ├── config/
    ├── share/
    ├── home/
    └── bin/

Categories

Dotman automatically categorizes files based on their destination path. Each category maps to a specific system location:

config/ → ~/.config/
share/ → ~/.local/share/
home/ → ~/
local/ → ~/.local/
bin/ → ~/.local/bin/

When adding files, dotman automatically places them in the correct category. For example:

dotman add ~/.config/nvim # Stored in config/
dotman add ~/.bashrc # Stored in home/ (as bashrc)
dotman add ~/.local/share/fonts # Stored in share/

Link States

The status command shows the state of each file in your profiles:

  • Linked - The file is properly symlinked from the profile to the system
  • NotLinked - The file exists in the profile but is not symlinked
  • Conflict - A file exists at the target location but is not managed by dotman
  • OtherProfile - The file is linked to another profile

Profiles

Profiles allow you to manage different configurations for different environments (e.g., work, personal, main). The main profile is created by default and cannot be deleted.

# Create a new profile
dotman make work

# Clone an existing profile
dotman clone main personal

# List all profiles
dotman list

# Delete a profile (not main)
dotman --delete-profile work

Validation

Dotman automatically validates symlinks before creating them to prevent conflicts and data loss:

  • Checks for existing files at the target location
  • Detects conflicts with other profiles
  • Validates directory structure
  • Reports broken or orphaned symlinks

Profile Commands

init

Initialize dotman directory structure in ~/.dotfiles.

dotman init
# or using alias
dotman -i

make

Create a new profile with the specified name.

dotman make <name>
# or using alias
dotman -m <name>

clone

Clone an existing profile to create a new one.

dotman clone <source> <destination>
# or using alias
dotman -c <source> <destination>

list

List all available profiles.

dotman list
# or using alias
dotman -l

Manage Commands

set

Move a file or directory from the system to the current profile and create a symlink in its place. Use this when you want to start managing an existing configuration file.

dotman set <file>
# or using alias
dotman -s <file>

set vs add: Use set to move existing files from your system into dotman management. Use add when files already exist in the profile and you just want to create symlinks.

unset

Remove a symlink and restore the original file or directory from the profile back to its system location. This is the reverse operation of set.

dotman unset <file>
# or using alias
dotman -u <file>

Link Commands

add

Create a symlink from the profile to the system. The file must exist in the profile directory. Unlike set, this command does not move files from the system—it only creates symlinks.

dotman add <file>
# or using alias
dotman -a <file>

remove

Remove a symlink from the profile.

dotman remove <file>
# or using alias
dotman -r <file>

Status Commands

status

Display the status of all files in the current profile.

dotman status

Options:

  • --category <cat> - Filter by category (config, share, home, local, bin)
  • --linked - Show only linked files
  • --not-linked - Show only unlinked files
  • --conflicts - Show only conflicting files
  • --other - Show other profile files
  • --verbose, -v - Show detailed information
  • --profile <name> - Use specific profile
  • --ascii - Use ASCII characters in table

Examples:

dotman status --category config
dotman status --linked --verbose
dotman status --conflicts
dotman status --profile work --ascii

Sync Commands

push

Copy files from the system to the profile. Use this when you've manually edited a configuration file and want to update the profile.

dotman push <profile>

pull

Update files on the system from the profile. Use this when you've edited files in the profile directly (e.g., via Git).

dotman pull <profile>

Global Options

  • --profile <name> - Set active profile (default: main)
  • --delete-profile <name> - Delete a profile
  • --help, -h - Show help message
  • --version, -v - Show version information

Design Principles

Dotman is built with Data Oriented Design (DOD) principles for maximum performance:

  • Data > Code - Data structures are designed first for hardware efficiency
  • SoA over AoS - Structure of Arrays preferred for SIMD-friendly memory layout
  • Batch Processing - Functions operate on collections, not individual elements
  • Predictable Memory - Minimal GC allocations, pre-allocated buffers
  • Pure Functions - Functions without side effects where possible

This architecture ensures optimal CPU cache utilization and fast operations even with thousands of configuration files.

Tips & Best Practices

Use Git with Dotman

Version control your dotfiles with Git by initializing a repository in ~/.dotfiles:

cd ~/.dotfiles
git init
git add .
git commit -m "Initial commit"
git remote add origin <your-repo-url>
git push -u origin main

Handle Conflicts

If dotman detects a conflict (a file exists at the target location that isn't managed by dotman), it will report it in the status. Resolve conflicts by:

  • Moving or deleting the conflicting file manually
  • Using --conflicts flag to identify issues
  • Checking which profile manages the file with --other flag

Profile Organization

Keep your profiles organized by environment:

  • main - Common configurations used across all machines
  • work - Work-specific configurations (Git config, SSH keys, etc.)
  • personal - Personal configurations
  • server - Server-specific configurations

Sync Workflows

Use push and pull commands to keep your system and profiles in sync:

  • push - After manually editing a config file on your system, push changes back to the profile
  • pull - After pulling changes from Git, update your system from the profile

Examples

Neovim Configuration

Manage your Neovim setup across multiple machines:

# Move nvim config to profile and link it
dotman set ~/.config/nvim

# Check status of config category
dotman status --category config

# View detailed breakdown by subdirectory
dotman status --verbose

Font Management

Sync your fonts across multiple machines:

# Add fonts directory to profile
dotman set ~/.local/share/fonts/Monaspace

# Check status with filters
dotman status --category share --linked

# Check for any conflicts
dotman status --conflicts

Fish Shell Setup

Set up your Fish shell environment:

# Add fish configuration files to profile
dotman set ~/.config/fish/config.fish
dotman set ~/.config/fish/functions.fish

# Verify all fish files are properly linked
dotman status --category config --verbose

Multi-Profile Workflow

Manage separate configurations for different environments:

# Create profiles for different environments
dotman make work
dotman make personal

# Add work-specific config to work profile
dotman --profile work set ~/.config/git/config

# List all profiles
dotman list

# Check work profile status
dotman status --profile work

# Delete a profile (main cannot be deleted)
dotman --delete-profile work

Migration from Other Managers

From GNU Stow

To migrate from Stow, first remove existing symlinks, then move your dotfiles to the dotman structure:

# Remove Stow symlinks
stow -t $HOME --delete *

# Initialize dotman
dotman init

# Move your stow directory contents
mv ~/dotfiles/* ~/.dotfiles/main/

# Re-link with dotman
dotman add

From yadm

Dotman can read yadm-style repositories. Simply move your yadm repository to ~/.dotfiles and adjust the structure to match dotman's category-based organization.

Manual Setup

If you're coming from another dotfile manager or starting fresh, you can manually organize your files into dotman's structure:

# Initialize dotman
dotman init

# Manually organize files by category
mkdir ~/.dotfiles/main/config
mkdir ~/.dotfiles/main/share
mkdir ~/.dotfiles/main/home

# Copy your config files to appropriate categories
cp -r ~/.config/nvim ~/.dotfiles/main/config/
cp -r ~/.local/share/fonts ~/.dotfiles/main/share/
cp ~/.bashrc ~/.dotfiles/main/home/

# Create symlinks for all files in profile
dotman add nvim
dotman add fonts
dotman add bashrc
